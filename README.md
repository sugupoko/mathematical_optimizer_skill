# Mathematical Optimizer Skill Pack

> **注意**: これは数理最適化を勉強しながら作成したものです。合成データでの検証のみで、実務で使えるかはわかりません。「こういうアプローチがあるんだな」程度の参考としてご覧ください。

Claude Code で数理最適化を行うためのスキルパック。

工場のシフト表作成、Amazonの配送ルート最適化、Kaggle Santaコンペ2025（クリスマスツリーパッキング）の上位解法を参考に、**最適化スペシャリストの思考パターンを7つに抽出してスキル化**した。

「シフト表を最適化して」「配送ルートを効率化して」と言われた時に、**データを受け取ってから改善提案を出すまで**を再現性のある手順で進められる。

## こういう人向け

- 「最適化やりたいけど何から始めれば？」という人
- OR-Toolsは知っているが、問題の分析→改善→提案の流れが定まっていない人
- 現場からデータを受け取って、経営に提案書を出す立場の人

## 何ができるか

データと「最適化して」の一言で、ここまで自動で進みます:

1. 問題の種類を分類（シフト？配送？割当？）
2. ベースラインを構築し、ボトルネックを特定
3. 改善策を設計・検証
4. 経営向けの改善提案書を生成
5. 足りないデータがあれば依頼書も生成

## クイックスタート — 3ステップで動く

### ① データを置く

`workspace/my_project/data/` にファイルを入れるだけ。
ヒアリングした内容や、既存のシフト表など。Excel でも CSV でも OK。
これからヒアリングする場合は `reference/hearing_templates.md` を参照。

```bash
mkdir -p workspace/my_project/data
cp あなたのデータ.xlsx workspace/my_project/data/
```

### ② Claude Code に指示する

```
workspace/my_project/data/ にあるデータで、最適化して
```

これだけ。あとは Claude が問題の種類を判断して、分析→ベースライン→改善まで進めます。

> **もっと細かく制御したい場合** は `/opt-assess`, `/opt-baseline` 等のスキルを個別に呼べます（→[使い方](#使い方)を参照）

### ③ 結果を見る

完了すると以下が生成されます:

```
workspace/my_project/
├── reports/    ← 提案書・分析レポート（これをクライアントに渡す）
├── results/    ← 数値結果（JSON）
└── scripts/    ← 実行スクリプト（再実行・カスタマイズ用）
```

### ML × 最適化の組合せ（発展的な使い方）

数理最適化は「パラメータが確定していれば最適解を出せる」。
でも実務では「そのパラメータが不確実」なことが多い。そこでMLと組み合わせる。

```
パターン1: ML予測 → 最適化       （一番よく使う）
  需要予測→シフト人数、配送時間予測→ルート、故障予測→保守スケジュール

パターン2: 最適化 → ML評価       （現場受容性の向上）
  ソルバーが複数案生成 → MLが「現場で却下されそうな案」を除外

パターン3: MLが目的関数を学習     （暗黙知の自動化）
  ベテランの修正パターンから「何が良い解か」をMLが学習 → 目的関数に反映

パターン4: ML ←→ 最適化ループ    （アルゴリズム自体の改善）
  LLMがヒューリスティクスを設計 → 実行 → 分析 → 改良（FunSearch/ReEvo方式）
```

詳しくは `reference/ml_optimization_guide.md` を参照。

## 5つの原則

1. **まずソルバーで解けるか試す** — 5分でベースラインを作る
2. **評価関数を先に読む** — 何を最適化するか理解してからコードを書く
3. **目的関数と評価関数を一致させる** — これだけで+15-27%改善
4. **仮定を常に明示する** — 仮定が間違えば結果も間違う
5. **不可能なら不可能と言う** — 最も価値のある提言になり得る

## 初心者の方へ — AI先生(Claude)からのアドバイス

> テンプレートとガイドは揃いました。でも、**本当に大事なのはここからです。**

このスキルパックは「道具箱」であって「答え」ではありません。
実際のデータで回してみると、必ずこういうことが起きます:

```
❶ 「この制約、実はソフトだった」
   → 現場の人に聞いたら「できれば守りたいレベル」だった
   → ハード制約からソフト制約に変更し、重みを調整する

❷ 「この仮定が間違っていた」
   → 移動速度30km/hと仮定したが、実測は22km/hだった
   → 仮定を実データで置き換え、/opt-baseline からやり直す

❸ 「もっと大きい規模だった」
   → サンプルの20件では動いたが、実データの2000件で遅い
   → 分解戦略（クラスタ分割、AM/PM分割等）を選択する

❹ 「最適化の結果を現場が使ってくれない」
   → 数学的に最適でも、暗黙のルールを見落としていた
   → ヒアリングシートで暗黙知を引き出し、制約に反映する
```

これは失敗ではなく、**最適化の正常なサイクル**です。

```
  データを受け取る → 仮定を置く → 解く → 結果を見せる
       ↑                                    ↓
       └──── 仮定を修正する ← フィードバック ←┘
```

このサイクルを**速く回すこと**が、最適化で成果を出すコツです。
スキルパックは「1周目を5分で回す」ための道具。
2周目以降は、現場のフィードバックがあなたの最高の教科書になります。

**まずはサンプルデータで1周回してみてください。**
`workspace/examples/` の6つのサンプルが、それぞれ違う種類の問題を扱っています。
