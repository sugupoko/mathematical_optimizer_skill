# Skill: opt-baseline（ベースライン構築+ボトルネック分析）

`/opt-baseline [data_path]` で実行。
最速でベースラインを作り、「何が壁か」を特定する。

## いつ使うか
- /opt-assess で問題の分類が済んだ後
- 「まず動くものを見たい」時

## 実行内容

### 1. データ読込みと前処理
```
- assessで特定したデータ形式に合わせてパース
- 欠損値の処理（仮の値を入れてまず動かす）
- 距離マトリクス/コストマトリクスの生成（必要なら）
```

### 2. 3つのベースラインを作る（思考回路②の準備）

**ベースライン1: ランダム（下限の確認）**
```python
# 無作為に割当/順序を決める
# 「何もしないとこれだけ悪い」の基準
```

**ベースライン2: 貪欲法（常識的な解）**
```python
# 問題に応じた簡単なルール:
#   スケジューリング → スキルが高い人から順に割当
#   VRP → 最近傍法
#   パッキング → 大きいものから順に詰める
```

**ベースライン3: ソルバー（性能の上限）**
```python
# OR-Toolsで定式化して解く
# 時間制限30秒、まず解けるか確認
```

### 3. エラー分析（思考回路②）
```
3つのベースラインの結果を比較:

              ランダム    貪欲法    ソルバー
feasible?     x(50違反)   x(5違反)  o(0違反)
スコア        100        300       500

→ 「ソルバーは解けるが貪欲法は5違反」
→ 違反の内訳: 制約Aが3件、制約Bが2件
→ ボトルネック = 制約A
```

### 4. ボトルネック診断
```
違反が1種類に集中？
  → YES: その制約がボトルネック
  → NO: 制約緩和分析（1つずつ外して効果を測定）

ソルバーのスコアと貪欲法の差は？
  → 大きい（2倍以上）: 目的関数の設計に改善余地大
  → 小さい（20%以内）: 制約充足が主課題
```

### 5. 出力

バージョンフォルダ内に以下を保存する:
- `reports/baseline_report.md`（ベースライン結果・ボトルネック分析・次のステップ）
- `scripts/baseline.py`
- `results/baseline_results.json`

```markdown
## ベースライン結果

| 手法 | Feasible | 違反数 | スコア |
|------|----------|--------|--------|
| ランダム | ... | ... | ... |
| 貪欲法 | ... | ... | ... |
| ソルバー | ... | ... | ... |

## ボトルネック分析
- 最もきつい制約: [XX]（違反の80%を占める）
- ソルバーと貪欲法の差: [XX%]
- 改善余地: [大/中/小]

## 追加で欲しい情報
- [ ] [ボトルネック解消に必要な情報]

## 次のステップ
→ /opt-improve でボトルネックに対策
```

## 状態管理

### 読み込み
- バージョンフォルダ内の `.opt_state.yaml` の `assess` セクションを読み込む
- 問題の分類、仮定、データ品質の情報を引き継ぐ

### 書き込み
- 実行完了時にバージョンフォルダ内の `.opt_state.yaml` の `baseline` セクションを書き込む
- 3つのベースライン結果、ボトルネック分析、改善余地を記録
- スキーマは `reference/state_schema.md` を参照

## トラブルシューティング

### OR-Tools がインストールできない
```
症状: pip install ortools が失敗する
原因と対策:
  ├── 企業プロキシ → pip install --proxy http://proxy:port ortools
  ├── Python バージョン非対応 → OR-Tools は Python 3.8-3.12 対応。バージョン確認
  ├── ARM Mac (M1/M2) → pip install ortools で自動的にARM版が入る（9.7+）
  └── オフライン環境 → 別マシンで pip download ortools して wheel を持ち込む

代替手段:
  ├── PuLP + HiGHS（pip install pulp、プロキシでも比較的通りやすい）
  └── 貪欲法 + SWO+IFS（ソルバー不要、Python標準ライブラリのみ）
```

### ソルバーが時間内に解を返さない
```
症状: CP-SAT が time_limit 内に FEASIBLE にならない
対策:
  ├── time_limit を延長（30s → 120s → 300s）
  ├── 問題を分解する（思考回路①: AM/PM分割、クラスタ分割）
  ├── 変数を減らす（思考回路③: 明らかな割当を先に固定）
  ├── 制約を緩和してまず feasible を見つける
  │   → model.Proto().ClearField('objective') で目的関数を外す
  │   → feasible が見つかれば add_hint() で warm start
  └── num_workers を増やす（CPU コア数を指定）
```

### ベースラインが全て infeasible
```
症状: ランダムも貪欲法もソルバーも制約を満たせない
対策:
  ├── 問題そのものが不可能かもしれない → 思考回路⑦（不可能性の証明）
  ├── 制約の定式化が間違っている可能性 → 制約を1つずつ追加して確認
  ├── データに誤りがある可能性 → 制約値（上限/下限）の妥当性を確認
  └── ソルバーの定式化をデバッグ:
        1. 全制約を外して解けるか確認
        2. 制約を1つずつ追加
        3. 追加した瞬間に解けなくなる制約がボトルネック
```

### 距離マトリクスの生成でメモリ不足
```
症状: N×N の距離行列が大きすぎる（N>1000）
対策:
  ├── 疎行列で保持（k-nearest neighbor のみ計算）
  ├── int16 で保持（距離を10m単位にスケーリング）
  ├── オンデマンド計算（コールバック内で都度計算）
  └── reference/data_preprocessing.md の「大規模距離行列」セクションを参照
```
