# Skill: opt-baseline（ベースライン構築+ボトルネック分析）

`/opt-baseline [data_path]` で実行。
最速でベースラインを作り、「何が壁か」を特定する。

## いつ使うか
- /opt-assess で問題の分類が済んだ後
- 「まず動くものを見たい」時

## 実行内容

### 1. データ読込みと前処理
```
- assessで特定したデータ形式に合わせてパース
- 欠損値の処理（仮の値を入れてまず動かす）
- 距離マトリクス/コストマトリクスの生成（必要なら）
```

### 2. 3つのベースラインを作る（思考回路②の準備）

**ベースライン1: ランダム（下限の確認）**
```python
# 無作為に割当/順序を決める
# 「何もしないとこれだけ悪い」の基準
```

**ベースライン2: 貪欲法（常識的な解）**
```python
# 問題に応じた簡単なルール:
#   スケジューリング → スキルが高い人から順に割当
#   VRP → 最近傍法
#   パッキング → 大きいものから順に詰める
```

**ベースライン3: ソルバー（性能の上限）**
```python
# OR-Toolsで定式化して解く
# 時間制限30秒、まず解けるか確認
```

### 3. エラー分析（思考回路②）
```
3つのベースラインの結果を比較:

              ランダム    貪欲法    ソルバー
feasible?     x(50違反)   x(5違反)  o(0違反)
スコア        100        300       500

→ 「ソルバーは解けるが貪欲法は5違反」
→ 違反の内訳: 制約Aが3件、制約Bが2件
→ ボトルネック = 制約A
```

### 4. ボトルネック診断
```
違反が1種類に集中？
  → YES: その制約がボトルネック
  → NO: 制約緩和分析（1つずつ外して効果を測定）

ソルバーのスコアと貪欲法の差は？
  → 大きい（2倍以上）: 目的関数の設計に改善余地大
  → 小さい（20%以内）: 制約充足が主課題
```

### 5. 出力
```markdown
## ベースライン結果

| 手法 | Feasible | 違反数 | スコア |
|------|----------|--------|--------|
| ランダム | ... | ... | ... |
| 貪欲法 | ... | ... | ... |
| ソルバー | ... | ... | ... |

## ボトルネック分析
- 最もきつい制約: [XX]（違反の80%を占める）
- ソルバーと貪欲法の差: [XX%]
- 改善余地: [大/中/小]

## 追加で欲しい情報
- [ ] [ボトルネック解消に必要な情報]

## 次のステップ
→ /opt-improve でボトルネックに対策
```
