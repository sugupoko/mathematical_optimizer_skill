# Baseline Report — worker_supervisor v1

**日付**: 2026-04-11
**戦略**: 段階的解法 (complexity=complex)
**ソルバー**: OR-Tools CP-SAT, time_limit=60s/phase, num_workers=4

## 概要

20 作業者 + 8 監督者 × 42 シフト (2週間) = **1,176 バイナリ変数**の複合スケジューリング問題。
12 個の HC を 7 Phase で段階的に追加し、どこで壁にぶつかるか特定。

## Phase 別結果

### 読み方

各 Phase でソルバーに渡す制約を段階的に増やす。独立検証器は**全 12 HC** をチェックするが、結果を 2 つに分けて報告:

- **active**: その Phase で**すでに強制している** HC。ソルバーが正しければ違反ゼロになるはず → ゼロでなければ実装バグ
- **pending**: まだモデルに入れていない HC。違反があっても OK (後の Phase で解消する対象)

**壁の定義**: ある Phase で**ソルバーが INFEASIBLE** を返し、直前の Phase では feasible だった → その Phase で追加した HC が既存 HC と矛盾している

### Phase 表

| Phase | 追加 | ソルバー | w割当 | v割当 | active HC 違反 | pending 違反 (Δ) | 時間 |
|-------|------|:---:|-----:|-----:|:---:|---:|---:|
| 0 | - | OPTIMAL | 0 | 0 | ✓ (制約なし) | 88 | 0.04s |
| 1 | HC1 | OPTIMAL | 160 | 0 | ✓ | 135 (Δ +47) | 0.01s |
| 2 | HC2 | OPTIMAL | 160 | 46 | ✓ | 116 (Δ -19) | 0.01s |
| 3 | HC3, HC4 | OPTIMAL | 160 | 46 | ✓ | 88 (Δ -28) | 0.05s |
| 4 | HC5, HC6 | OPTIMAL | 160 | 46 | ✓ | 70 (Δ -18) | 0.03s |
| **5** | **HC7, HC8** | **INFEASIBLE** | - | - | - | - | 0.01s |
| 6 | HC9, HC10 | INFEASIBLE | - | - | - | - | 0.01s |
| 7 | HC11, HC12 | INFEASIBLE | - | - | - | - | 0.01s |

### 観察

1. **Phase 0-4 は全て active OK**: 制約追加のたびにソルバーは新しい active 制約を完全に満たす解を見つけている
2. **Pending は単調減少していない** (Phase 1 → 2 で +19 増加): 制約を追加するとソルバーの最適化方向が変わり、副作用で他の未適用 HC がより多く違反する方向に動くことがある (これは想定通りの挙動)
3. **Phase 5 で初めて INFEASIBLE**: Phase 4 まで feasible で、Phase 5 で HC7, HC8 を追加した瞬間に解が存在しなくなった
4. **Phase 6, 7 は Phase 5 の infeasibility を引き継いだだけ** (HC9-HC12 は無実を意味する)

**→ 壁: Phase 5 で追加した HC7 (skill) + HC8 (bilingual) の組合せ**

## 根本原因分析

### 壁にぶつかった制約: HC7 (skill) + HC8 (bilingual)

データから候補・需要を計数した結果、**鳩の巣原理による構造的不可能性**を発見:

#### 原因 1: bilingual + reception シフトの人員不足

**事実** (R1, R3 から):
- 作業者で en (英語) を話せる人: **6 名** (W002, W004, W008, W010, W013, W017)
- このうち `reception` スキルを持つ人: **4 名** (W002, W004, W008, W010)
- `bilingual_required=yes` かつ `required_skills=reception` のシフト: **4 シフト** (各週の Tue/Thu morning)
- 各シフトは `worker_required = 5` 名

**不可能性の証明** (鳩の巣原理):
```
1 シフトに必要: 5 名
配置可能な候補プール: 4 名 (en ∩ reception)
→ 5 > 4 → どんなアルゴリズムを使っても絶対に配置不可能
```

これは「工夫すれば解ける」問題ではなく、**プール < 需要** の数学的不可能性。

#### 原因 2: bilingual + phone シフトはギリギリ

- 作業者で en + phone を持つ人: **6 名** (全 en 作業者)
- `bilingual_required=yes` かつ `required_skills=phone` のシフト: **4 シフト** (各週の Thu/Fri afternoon) × 各5名
- 数的には 6 ≥ 5 なので「1 シフト単体では」満たせる
- ただし 2週間で **20 worker-shifts 必要** に対し、6 名 × 14 日 ≒ 84 日分しか無い
- 実際には作業者の max_hours, 不可日, HC9 休息制約で更に絞られる
- → 原因1 で既に infeasible なので、これは単独の原因にはならないが、解決時に注意が必要

### 監督者側の確認

- 監督者で en を話せる人: **4 名** (8 名中 — S001, S003, S005, S008)
- bilingual 全 8 シフトで監督者需要は 1-2 名
- 監督者側は余裕あり (4 名で十分)

### 無実の制約 (容疑から除外)

段階的に追加したことで以下が無実と判明:
- **HC9, HC10 (休息)**: Phase 5 で既に壁なので影響なし
- **HC11 (forbidden ペア)**: 2件のみ、影響小
- **HC12 (mentorship ペア)**: 4件、2週間での同時シフト達成は容易

→ **ボトルネックは純粋に HC7 ∩ HC8 の組合せ**。他の複雑な制約は原因ではない。

## 解決策の選択肢

### A. 入力を変える (データ側)

| 案 | 内容 | 追加コスト | 導入難易度 |
|---|---|---|---|
| **A1. 英語+reception 作業者を1名追加** | en + reception スキルを持つ作業者を採用 (mid以上、max 40h) | 人件費 (中) | 採用次第 (1-2ヶ月) |
| A2. 既存作業者に英語研修 | 非英語作業者に ESL プログラム | 研修費 + 時間 | 3-6ヶ月 |
| A3. reception に新人を追加研修 | en を持つが reception 未習得の W013, W017 に reception 研修 | 研修費 | 1-2ヶ月 |

### B. 仕様を変える (HC の緩和)

| 案 | 内容 | 追加コスト | 導入難易度 |
|---|---|---|---|
| **B1. HC1 緩和**: bilingual+reception シフトを 5→4 名 | 英語顧客対応のキャパが 20% 減 | 0 円 | 顧客合意が必要 |
| B2. HC8 緩和: bilingual シフトに非英語可の配置を許容 | 非英語作業者を bilingual シフトに入れる | 0 円 | サービス品質合意 |
| B3. HC8 の解釈変更: 「全員en必須」→「最低1名en」 | 仕様の意味を緩める (spec Q5 で確認) | 0 円 | クライアント確認必須 |

### C. 運用で回避 (プロセス変更)

| 案 | 内容 | 追加コスト | 導入難易度 |
|---|---|---|---|
| **C1. bilingual 需要の再配置** | 英語顧客対応時間を Tue/Thu morning → 別時間帯に分散 | 顧客導線変更 | 顧客との調整必要 |
| C2. 外注/一時派遣 | bilingual 対応を外部スタッフで週 4 シフトだけ補う | 外注費 | 契約締結のみ |

### 推奨組合せ

- **即時 (今週中)**: **B1** で暫定運用 (bilingual+reception シフトは 4 名で回す)
- **中期 (1-2ヶ月)**: **A1** の採用を並行して進める (恒久対策)
- **クライアント確認**: assess の Q5 (bilingual_required の正確な意味) を確認 → もし B3 が成立すればコストゼロで解決可能

## 段階的アプローチの価値

**もし一発解きなら**: 12 個の制約を全部入れて INFEASIBLE と返るだけで、どの制約が原因か分からない。ペア制約 (HC11/HC12) や休息制約 (HC9/HC10) を疑って時間を浪費する可能性。

**段階化により判明したこと**:
1. HC1-HC6 (基礎需要・供給・可用性) は **完全に通る** (160/160, 46/46)
2. Phase 5 で HC7+HC8 を追加した瞬間に壁
3. → 原因は **スキル + 言語の組合せ問題のみ**
4. HC9-HC12 は無実
5. 切り分け時間: 全 Phase 合計 **0.12 秒** で原因特定
