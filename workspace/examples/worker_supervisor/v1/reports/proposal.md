# シフト最適化 改善提案書 — worker_supervisor v1

**作成日**: 2026-04-11
**対象**: 経営層 / 現場運用責任者
**プロジェクト**: 作業者 20 名 × 監督者 8 名 × 2 週間 (42 シフト) のスケジューリング

---

## 1. 現状の課題 — 数学的に「組めないシフト」が存在する

頂いたデータを最適化エンジン (Google OR-Tools CP-SAT) で精査した結果、
**現行の制約条件のままでは、どんな天才スケジューラーでも 100% 満たせないシフトが 4 つ存在する**ことが判明しました。

### 該当シフト

| シフト | 必要人数 | 必須条件 |
|---|---|---|
| 第1週 火曜 morning | 5 名 | reception スキル + 英語可 |
| 第1週 木曜 morning | 5 名 | reception スキル + 英語可 |
| 第2週 火曜 morning | 5 名 | reception スキル + 英語可 |
| 第2週 木曜 morning | 5 名 | reception スキル + 英語可 |

### 数学的な事実

- **必要**: 各シフト 5 名 × 4 シフト = 延べ **20 人枠**
- **供給**: reception スキル + 英語可の作業者は **4 名のみ** (W002 佐藤, W004 高橋, W008 中村, W010 加藤)
- **1 シフトあたりの供給上限 = 4 名** ← この時点で「5 名要求」は満たせない

これは「うまく組めば何とかなる」問題ではなく、**プールサイズ < 必要人数** という構造的な不可能性です。

### 数学的不可能性の証明

```
任意の 1 つの bilingual + reception シフト s について:
  HC1 (要求): Σ_w worker_x[w,s] = 5
  HC7 (skill): worker_x[w,s] = 0  (w が reception スキルを持たない場合)
  HC8 (lang): worker_x[w,s] = 0  (w が en を話せない場合)

→ 候補 w は {W002, W004, W008, W010} の 4 名のみ
→ Σ ≤ 4 < 5  ⇒ 矛盾
```

ベースライン段階 (`reports/baseline_report.md`) で 7 段階に分けて 12 個のハード制約を逐次追加した結果、
Phase 1-4 (基礎需要・供給・休暇) は問題なく解け、**Phase 5 で HC7 (スキル) と HC8 (英語) を追加した瞬間に infeasible**。
4 シフト合計で **延べ 4 名分の不足** が発生していることを確認しました。

## 2. 改善策 — 3 つの選択肢

| 案 | 内容 | HC充足 | コスト | 影響 | 推奨度 |
|---|---|:---:|---|---|:---:|
| **A. 採用案 (恒久対策)** | 英語可 + reception スキルの中堅作業者を **1 名採用** (W021) | ✅ **全12HC充足** | 新規人件費 | サービス品質維持、2 週間で安定運用 | **◎** |
| **B. 運用妥協案** | 4 つの bilingual+reception シフトを 5 名 → 4 名で運用 (HC1 を緩和) | ❌ **HC1 を 4 件緩和** | **0 円** | 英語顧客対応のキャパが 20% 減 | △ |
| **C. シフト再設計** | 英語顧客対応の需要を 別曜日 (Mon/Wed) や afternoon に分散 | - (要再検証) | 顧客導線の変更 | 顧客側オペ変更必要、即時実行不可 | △ |

**独立な HC 検証器** で A, B の結果を再チェックしました:

- **A (W021 採用)**: 全 12 個のハード制約 (HC1-HC12) をすべて満たす唯一の解
- **B (HC1 緩和)**: 4 つの bilingual+reception シフトで「5 名要求」を破っている (HC1 違反 4 件)。これは仕様書 (`spec.md`) の HC1 を変更することを意味する
- B, C はどちらも「元の仕様を緩めないと解けない」ことの数学的証明であり、A が**唯一、仕様通りに組める案**

### 数値的検証 (シナリオ A: W021 採用)

| 指標 | 値 |
|---|---|
| 全 12 ハード制約 | ✅ **すべて充足 (独立検証済み)** |
| HC 違反 | 0 件 |
| ソフト制約 総合スコア (0-100) | **83.0** |
| 連続勤務 ≤ 5 日 (SC1) | 100 (完全達成) |
| 作業者労働時間バラツキ (SC2) | 80 (16h 差) |
| 監督者労働時間バラツキ (SC3) | 90 (8h 差) |
| 作業者夜勤 ≤ 4 (SC4) | 100 |
| 監督者夜勤 ≤ 3 (SC5) | 100 |
| 最低勤務時間 (SC6) | 52 (構造的に難) |
| シニア配置 ≥ 1/日 (SC7) | 100 |
| 推奨ペア達成 (SC8) | 42 (16 回成立) |

**重要**: シナリオ B の「ソルバー feasible=True」は**緩和後モデル**でのものであり、
元の spec.md の HC1 に対しては 4 件の違反を抱えています。
B を採用する場合は、spec.md の HC1 を正式に修正する必要があります。

### 想定採用要件 (W021)

| 項目 | 推奨値 |
|---|---|
| スキル | reception, phone, chat |
| 言語 | 日本語 + 英語 |
| レベル | mid (経験 3-5 年) |

## 3. ソフト制約バリエーション (5 案の比較)

5 通りの「優先順位」で再最適化し、どの SC が動かせて、どの SC が構造的下限なのかを検証しました。

| 重み案 | 強調 SC | 平均スコア | 主な変化 |
|---|---|---:|---|
| V1 balanced | (基準) | 83.0 | デフォルト |
| V2 fairness max | SC2+SC3 (公平性) | 83.0 | obj 増 (重み倍化のため) だが raw は不変 |
| V3 coverage max | SC1+SC6 (カバレッジ) | 83.0 | 同上 |
| V4 welfare | SC4+SC5 (夜勤上限) | 83.0 | 同上 |
| V5 senior coverage | SC7 | 83.0 | 同上 |

### 重要な発見

5 案で **SC raw 値が完全に一致** しました:
- SC2 spread = 16h, SC3 spread = 8h, SC6 shortfall = 96h, SC8 = 16 ペア成立

これは「重みが効かない」のではなく:
1. **time_limit 60 秒では巨大な解空間 (1,176 バイナリ変数) を探索しきれない**
2. **SC2/SC6 の値は構造的な下限 (W008/W016 の上限 24h など) で決まっている可能性が高い**

→ v2 で改善する余地: 探索時間延長 (300-600s) + 段階的最適化 (lex 法) を試す。

## 4. 納品形態の提案

最終納品物は **Excel 1 ファイル** を推奨 (shift_scheduling 案件と同形式)。

| シート | 内容 |
|---|---|
| `shift_table` | 14 日 × 3 シフトのマトリクス、セルに作業者ID/監督者ID |
| `worker_summary` | 作業者別の総時間、夜勤回数、連続勤務日数、最低時間達成チェック |
| `supervisor_summary` | 監督者別の同上 |
| `pair_check` | 禁止ペア違反 (= 0 を確認)、メンタルペア達成回数 (≥ 2 を確認) |
| `bilingual_check` | bilingual_required シフトの英語可比率 (B 案では 4/5、A 案では 5/5) |
| `kpi_dashboard` | SC1-SC8 の達成度ダッシュボード (0-100) |
| `assumptions` | spec.md A1-A10 の仮定一覧 (現場と摺り合わせ用) |
| `change_log` | v1→v2 の変更履歴 (採用/制約変更等) |

加えて生成物として:
- `proposal.pdf` (この提案書の PDF 版)
- `model_spec.md` (技術担当向けの数式仕様)
- `run.py` + `requirements.txt` (再実行可能なスクリプト一式)
