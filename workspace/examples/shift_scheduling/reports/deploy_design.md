# シフト最適化 — 運用設計書

**作成日**: 2026-03-30
**対象**: 10名体制シフトスケジューリング（提案A採用前提）

---

## 1. 実行概要

| 項目 | 設定 |
|------|------|
| 実行頻度 | **週次**（毎週金曜17:00） |
| 計算時間制限 | 30秒（実測0.05秒で完了） |
| データソース | CSV手動配置（`data/` フォルダ） |
| 出力 | CSV（`results/schedule_YYYYMMDD_HHMMSS.csv`） |
| 実行スクリプト | `scripts/run_weekly.py` |

### cron 設定

```bash
# 毎週金曜17:00に翌週のシフト表を生成
0 17 * * 5 cd /path/to/shift_scheduling && python scripts/run_weekly.py
```

---

## 2. パイプライン構成

```
Step 1: データ取得（CSV読み込み + 需要調整の適用）
  ↓
Step 2: バリデーション（致命的問題があれば中断）
  ↓
Step 3: 前処理（CSVパース時に完了）
  ↓
Step 4: CP-SAT最適化（time_limit=30s）
  ↓
Step 5: 結果検証（feasibility + 異常検知）
  ↓
Step 6: CSV出力 + ログ保存
```

### 需要調整（提案Aの反映）

パイプライン内で自動適用:
- **水曜朝勤**: 3名 → 2名
- **日曜午後勤**: 2名 → 1名

変更箇所: `run_weekly.py` の `DEMAND_OVERRIDES` 定数

---

## 3. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| 従業員数 | 0名 | **中断**（致命的） |
| 従業員数 | 8名未満 | 警告 |
| reception可能者 | 5名未満 | **中断**（致命的） |
| 供給 < 需要 | 1シフトでも不足 | **中断**（致命的） |
| max_hours > 60h | — | 警告（労基法超過の可能性） |
| min_hours > max_hours | — | **中断**（データ不整合） |
| シフト数 | 21以外 | 警告 |

## 4. 監視指標

| 指標 | 正常範囲 | アラート条件 | 通知先 |
|------|---------|------------|--------|
| Feasibility | 常にtrue | falseになったら | シフト管理者 |
| 計算時間 | < 1秒 | 10秒超過 | システム担当 |
| 公平性（std） | < 8h | 10h超過 | シフト管理者 |
| 勤務時間レンジ | 24-40h | 16h以下の人が出たら | シフト管理者 |
| バリデーションエラー | 0件 | 1件以上 | データ入力担当 |

### 監視方法

- **ログファイル**: `log/run_YYYYMMDD_HHMMSS.log` に全ステップの結果を記録
- **メタデータ**: `results/meta_YYYYMMDD_HHMMSS.json` に数値指標を保存
- 異常検知: ログに `[ERROR]` があれば通知（メール or Slack webhook）

## 5. フォールバック手順

### パターン1: ソルバーがINFEASIBLE

```
原因: 従業員の変更（退職・休職）で供給が需要を下回った
対応:
  1. バリデーションで「供給 < 需要」が検知される
  2. パイプラインが中断される（exit code 1）
  3. 対応選択肢:
     a. DEMAND_OVERRIDES を追加して需要をさらに削減
     b. パート/応援を手配して供給を増やす
     c. 前週のシフト表をそのまま使う（最終手段）
```

### パターン2: データファイルが見つからない

```
原因: CSVの配置忘れ、ファイル名変更
対応:
  1. FileNotFoundError でパイプラインが停止
  2. ログを確認して正しいファイルを配置
  3. 手動で再実行: python scripts/run_weekly.py
```

### パターン3: 結果の品質が悪い（公平性が低い等）

```
原因: 従業員の制約変更（不可日の増加等）
対応:
  1. メタデータの hours_std を確認
  2. std > 10h なら制約を見直し
  3. 必要なら /opt-improve でソフト制約の重みを調整
```

### ロールバック

```bash
# 過去8回分の結果を保持（HISTORY_KEEP=8）
ls results/schedule_*.csv  # 過去の結果一覧

# 前回の結果に戻す場合:
# 最新のschedule_*.csvを削除し、1つ前のファイルを使用
```

## 6. モデル見直しトリガー

以下のいずれかが発生したら `/opt-assess` で再アセスメント:

- [ ] 従業員の入退社（人数変更）
- [ ] スキル構成の変化（研修完了でreception可能者が増加等）
- [ ] 勤務条件の変更（週最大時間、不可日の変更）
- [ ] シフト構成の変更（新しいシフト帯の追加、必要人数の変更）
- [ ] 現場から「結果がおかしい」のフィードバックが2回以上
