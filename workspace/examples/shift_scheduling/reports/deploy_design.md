# シフト最適化 — 運用設計書

**作成日**: 2026-03-30
**更新日**: 2026-03-30
**対象**: 10名体制シフトスケジューリング（需要調整付き）

---

## 1. 実行概要

| 項目 | 設定 |
|------|------|
| 実行頻度 | **週次**（毎週金曜17:00） |
| 計算時間制限 | 30秒（実測0.05秒で完了） |
| データソース | CSV手動配置（`data/` フォルダ） |
| 出力 | CSV（`results/schedule_YYYYMMDD_HHMMSS.csv`）+ JSON（メタデータ） |
| 実行スクリプト | `scripts/run_pipeline.py` |

### cron 設定

```bash
# 毎週金曜17:00に翌週のシフト表を生成
0 17 * * 5 cd /path/to/shift_scheduling && python scripts/run_pipeline.py
```

## 2. パイプライン構成

```
Step 1: データ取得（CSV読み込み + 需要調整の自動適用）
  ↓
Step 2: バリデーション（致命的問題があれば中断）
  ↓
Step 3: CP-SAT改善版ソルバー実行（time_limit=30s）
  ↓  ※失敗時 → 貪欲法フォールバック
Step 4: 結果検証（HC違反・公平性・夜勤分布チェック）
  ↓
Step 5: CSV + JSONメタデータ出力 + ログ保存
```

### 需要調整（自動適用）

供給368h < 需要384hの構造的不足に対し、以下を `DEMAND_OVERRIDES` で自動適用:

| 曜日・シフト | 調整前 | 調整後 | 理由 |
|-------------|--------|--------|------|
| 水曜・朝勤 | 3名 | 2名 | 水曜は比較的業務が軽い |
| 日曜・午後勤 | 2名 | 1名 | 日曜午後は来客が少ない |

調整により需要が **384h → 368h** となり、理論上全シフト充足可能。ただし不可日・スキル制約により2枠程度の不足が残る可能性がある。

## 3. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| 従業員数 | 0名 | **中断**（致命的） |
| 従業員数 | 8名未満 | 警告 |
| reception可能者 | 5名未満 | **中断**（致命的） |
| 供給 < 需要（調整後） | 1hでも不足 | **中断**（致命的） |
| max_hours > 60h | — | 警告（労基法超過の可能性） |
| min_hours > max_hours | — | **中断**（データ不整合） |
| シフト数 | 21以外 | 警告 |

## 4. 最適化手法

**CP-SAT改善版**（solve_all.pyのcpsat_improved相当）:

- HC1（必要人数）: 緩和して不足ペナルティ(-200/枠)を目的関数に組込み
- HC2（週最大勤務時間）: ハード制約
- HC5（夜勤→朝勤禁止）: ハード制約
- SC1（連続勤務≤5日）: ペナルティ -10
- SC2（公平性）: 偏差ペナルティ -5/シフト
- SC3（最低勤務時間）: 不足ペナルティ -24/シフト
- SC4（夜勤≤2回/週）: 超過ペナルティ -8/回
- SC5（トレーニング配置）: ボーナス +5/日

### 期待される結果

| 指標 | 値 |
|------|-----|
| HC違反 | 0〜2枠（需要調整の効果による） |
| 夜勤3回以上 | 0名 |
| 時間偏差 | 5〜6h（構造的制約による下限） |
| 計算時間 | < 1秒 |

## 6. フォールバック手順

### パターン1: ソルバーがINFEASIBLE / 解なし

```
原因: 従業員の変更（退職・休職）で供給がさらに減少
対応:
  1. パイプラインが自動で貪欲法フォールバックを実行
  2. 貪欲法も失敗 → exit code 1でパイプライン停止
  3. 手動対応:
     a. DEMAND_OVERRIDES を追加して需要をさらに削減
     b. パート/応援を手配して供給を増やす
     c. 前週のシフト表をそのまま使う（最終手段）
```

### パターン2: データファイルが見つからない

```
原因: CSVの配置忘れ、ファイル名変更
対応:
  1. FileNotFoundError でパイプラインが停止
  2. ログを確認して正しいファイルを配置
  3. 手動で再実行: python scripts/run_pipeline.py
```

### パターン3: 結果の品質が悪い（公平性が低い等）

```
原因: 従業員の制約変更（不可日の増加等）
対応:
  1. メタデータの hours_std を確認
  2. std > 10h なら制約を見直し
  3. 必要なら /opt-improve でソフト制約の重みを調整
```

### ロールバック

```bash
# 前回の結果に戻す場合:
# 最新のschedule_*.csvを削除し、1つ前のファイルを使用
```
