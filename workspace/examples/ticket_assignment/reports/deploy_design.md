## 2. パイプライン構成

```
Step 1: データ取得（engineers/tickets/resolution_history CSV）
  ↓
Step 2: バリデーション（エンジニア勤務状況・P1未対応チェック）
  ↓
Step 3: 停滞検出（6h超 + 進捗50%未満のチケットを特定）
  ↓
Step 4: CP-SAT複合最適化（ブロック解放 + 停滞再アサイン）
  ↓  ※失敗時 → 貪欲法フォールバック
Step 5: 結果検証（HC違反・アサイン率・優先度別カバー率）
  ↓
Step 6: アサインJSON出力
```

### 複合改善の仕組み

3つの施策を同時に実行:

1. **ブロック解放**: ブロック中チケット（ベンダー待ち・承認待ち等）のスロットを解放
2. **停滞再アサイン**: 停滞チケットを検出し、別のエンジニアへ振り替え
3. **最適アサイン**: CP-SATで未アサイン + 停滞チケットを一括最適化

---

## 3. 停滞検出ルール

| 条件 | 閾値 | 判定 |
|------|------|------|
| ステータス | in_progress | 対象 |
| 経過時間 | > 6時間 | 必須 |
| 実進捗 / 期待進捗 | < 50% | 必須 |
| ブロック中 | blocked_* | **除外**（外部要因） |

期待進捗 = 経過時間 / (経過時間 + 推定残り時間) × 100%

---

## 4. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| エンジニアデータ空 | 0名 | **中断** |
| チケットデータ空 | 0件 | **中断** |
| 勤務中エンジニア | 0名 | **中断** |
| P1未アサイン | 1件以上 | 警告（即時対応必要） |
| SLAリスク(<6h) | 1件以上 | 警告 |

## 5. スコアリング

| 要素 | 配点 | ロジック |
|------|------|---------|
| スキル一致 | 30点 | required_skill が engineer.skills に含まれるか |
| SLA緊急度 | 30点 | 残り時間が少ないほど高スコア |
| 経験年数 | 15点 | 年数/15 × 15（最大15年でキャップ） |
| ティア適合 | 10点 | 必要ティアと一致で10点、1段上で5点 |
| LLM信頼度 | 5点 | 個人実績あり=5、カテゴリ平均=3、デフォルト=1 |

優先度ボーナス: P1=+5000, P2=+3000, P3=+1000, P4=0

### L1/L2/L3ティア制約

| ティア | 対応可能チケット種別 |
|--------|-------------------|
| L1 | service_request, incident_low |
| L2 | L1 + incident_mid, change_standard |
| L3 | L2 + incident_high, incident_critical, change_emergency |

- L3エンジニアはP1/Critical案件に温存
- off-shiftのL3エンジニアはP1のみon_callで対応可能

## 8. フォールバック手順

### パターン1: ソルバーが解なし

```
原因: 全エンジニアのスロットが埋まっている
対応:
  1. パイプラインが自動で貪欲法フォールバックを実行
  2. 貪欲法: 優先度 × SLA緊急度でソートし、ベストスコアのエンジニアに割当
  3. P1が未アサインのまま残った場合 → 即時エスカレーション
```

### パターン2: 全スロット枯渇

```
原因: エンジニア不足 or ブロック中チケットが大量
対応:
  1. ブロック中チケットの解放状況を確認
  2. オフシフトのL3エンジニアにP1のみon_callで依頼
  3. 停滞チケットの強制再アサインを検討
  4. 人員追加をエスカレーション
```

### パターン3: P1チケットの即時対応

```
対応:
  1. P1検知時に即時トリガーでパイプラインを実行
  2. P1に+5000ボーナスにより最優先でアサイン
  3. 必要ならオフシフトL3エンジニアをon_callで呼び出し
```

### ロールバック

```bash
# 過去96回分の結果を保持（15分×96=24時間分）
ls results/assignments_*.json
```

## 9. LLMエスティメータについて

解決時間の予測に3段階フォールバック方式を採用:

1. **個人実績**: 当該エンジニアの同一スキルでの過去平均解決時間
2. **カテゴリ平均**: 全エンジニアの同一スキルでの平均解決時間
3. **デフォルト**: 2.0時間（データがない場合）

将来的には:
- 実際のLLM（GPT/Claude API）による自然言語チケット分析
- チケット本文から難易度・所要時間をより精密に予測
- フィードバックループで予測精度を継続的に改善
