## 1. 現状の課題

### 1.1 スロット枯渇
エンジニア20名のうち勤務中15名が既存チケットで手一杯。新規チケット25件のうち通常の方法では9件（36%）しかアサインできない。

### 1.2 ブロック中チケットによるスロット無駄遣い
ベンダー回答待ち・承認待ちなどの8件がエンジニアのスロットを占有。実際には作業できないにもかかわらず、容量を消費している。

### 1.3 停滞チケットの放置
33件のチケットで進捗が期待の50%未満。担当者が手を動かせていない（もしくは他の優先作業に追われている）状態が長時間続いている。

### 1.4 P1/P2チケットの未対応
**TK036（認証基盤停止・P1・SLA残4時間）**が未アサインのまま放置されている。これは全社的な業務停止リスクを意味する。

## 2. 提案する改善策

### 施策1: ブロック中チケットのスロット解放
- **内容**: ブロック中（vendor待ち・承認待ち・顧客待ち）のチケットを「保留」扱いとし、担当者のスロットを解放
- **効果**: +5件のアサイン（36% → 56%）
- **リスク**: ブロック解除時に再アサインが必要（自動化で対応可）
- **実施難易度**: 低（運用ルールの変更のみ）

### 施策2: 停滞チケットの再アサイン
- **内容**: 基準時刻から6時間以上経過し、進捗が期待の50%未満のチケットを別のエンジニアに再アサイン
- **効果**: 33件の停滞チケットのうち20件を再アサイン
- **リスク**: 引き継ぎコスト（既存の作業内容の共有が必要）
- **実施難易度**: 中（引き継ぎプロセスの整備が必要）

### 施策3: 施策1+2の同時実施（推奨）
- **効果**: アサイン率74.1%、P1全件対応、P2 6件追加対応
- **ハード制約違反**: 0件
- **スキル一致率**: 100%

## 4. 実装ロードマップ

### Phase 1（即時実行）: ブロック解放ルール導入
- ブロック中チケットのスロット解放ルールをチケット管理システムに反映
- 所要時間: 1日

### Phase 2（1週間以内）: 停滞検出の自動化
- 停滞判定基準（6h経過 + 進捗50%未満）をダッシュボード化
- 停滞チケットの自動アラートと再アサイン提案
- 所要時間: 3〜5日

### Phase 3（1ヶ月以内）: CP-SATソルバーの定期実行
- 1時間ごとにソルバーを実行し、最適アサインを提案
- 承認フロー付きの半自動化システム
- 所要時間: 2〜4週間

## 5. 使用手法

- **Google OR-Tools CP-SAT**: 制約充足ソルバーによる最適化
- **LLMエスティメータ**: 過去の解決実績から解決時間を予測（個人平均→カテゴリ平均→デフォルトのフォールバック）
- **停滞検出**: 経過時間と進捗率の乖離から自動判定
- **スコアリング**: skill_match(30) + sla_urgency(30) + experience(15) + tier_fit(10) + llm_confidence(5)の多軸評価

## 6. 注意事項

- ブロック中チケットの扱いは外部要因であり、停滞とは区別して管理すること
- 再アサインは引き継ぎコストを伴うため、P3/P4の低優先度チケットは無理に再アサインしない判断も有効
- L3エンジニアはP1/Critical案件に温存し、P3/P4へのアサインは最小限にすること
- 今回の分析はスナップショット（2026-04-01 14:00時点）であり、定期的な再実行を推奨
