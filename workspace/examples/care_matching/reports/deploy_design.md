# 介護マッチング最適化 — 運用設計書

**作成日**: 2026-03-30
**対象**: 15名の利用者 × 10名のヘルパー（東京都5区）

---

## 1. 実行概要

| 項目 | 設定 |
|------|------|
| 実行頻度 | **月次**（毎月1日）or 都度（利用者/ヘルパーの変更時） |
| 計算時間制限 | 60秒 |
| データソース | CSV手動配置（`data/` フォルダ） |
| 出力 | マッチング表CSV + メタデータJSON |
| 実行スクリプト | `scripts/run_pipeline.py` |

### cron 設定

```bash
# 毎月1日 09:00にマッチング最適化を実行
0 9 1 * * cd /path/to/care_matching && python scripts/run_pipeline.py
```

### 都度実行

```bash
# 利用者やヘルパーの追加・変更があった場合
python scripts/run_pipeline.py
```

## 2. パイプライン構成

```
Step 1: データ取得（care_receivers/caregivers/compatibility_history CSV）
  ↓
Step 2: バリデーション（受入容量・資格・実行可能ペア数チェック）
  ↓
Step 3: CP-SAT改善版ソルバー実行（time_limit=60s）
  ↓  ※失敗時 → 貪欲法フォールバック
Step 4: 結果検証（HC違反・互換性・性別一致・地区一致・継続維持・負荷分布）
  ↓
Step 5: マッチング表CSV + JSONメタデータ出力
```

## 3. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| 利用者データ空 | 0名 | **中断** |

| 受入容量 < 利用者数 | — | **中断** |

| 介護福祉士不足 | 必要数 > 供給数 | 警告 |
| 実行可能ペア数が少ない | < 利用者数 | 警告 |

## 4. 最適化手法

**CP-SAT改善版**（solve_all.pyのcpsat_improved相当）:

| 要素 | ボーナス/ペナルティ | 目的 |
|------|-------------------|------|
| 互換性スコア | ×100（整数化） | 基本品質 |
| マッチボーナス | +5000/人 | 全員マッチングを最優先 |
| 継続ボーナス（満足度5） | +3000/ペア | 信頼関係のあるペアを維持 |
| 性別一致ボーナス | +800/ペア | 利用者の性別希望を尊重 |
| 同一地区ボーナス | +600/ペア | 移動距離を短縮 |
| 隣接地区ボーナス | +300/ペア | 隣接区も許容 |
| 経験10年以上（要介護5） | +500/ペア | 重度利用者に熟練ヘルパー |
| 最大負荷ペナルティ | -1000×max_load | 負荷集中を抑制 |

## 6. フォールバック手順

### パターン1: ソルバーが全員未マッチ

```
原因: 制約が厳しすぎる（新しいNG制約の追加等）
対応:
  1. パイプラインが自動で貪欲法フォールバックを実行
  2. 貪欲法の結果を確認し、手動で調整
  3. 制約の緩和を検討（隣接区への配置許可等）
```

### パターン2: 新規利用者の追加

```
対応:
  1. care_receivers.csv に追加
  2. compatibility_history.csv に履歴があれば追加
  3. run_pipeline.py を再実行
  → 既存ペアは継続ボーナスで維持されつつ、新規利用者が最適に配置される
```

### パターン3: ヘルパーの休職・退職

```
対応:
  1. caregivers.csv から該当ヘルパーを削除（または max_clients を 0 に）
  2. run_pipeline.py を再実行
  3. 影響を受ける利用者は自動的に別のヘルパーに再配置
  4. 受入容量不足の場合はバリデーションで検知 → 採用対応
```

### ロールバック

```bash
# 過去6回分の結果を保持（HISTORY_KEEP=6）
ls results/matching_*.csv
# 前回の結果に戻す場合は該当CSVを参照
```
