# 配送ルート最適化 — 運用設計書

**作成日**: 2026-03-30
**更新日**: 2026-03-30
**対象**: 品川配送センター → 東京都内20顧客 × 3台（AM/PM 2便制）

---

## 1. 実行概要

| 項目 | 設定 |
|------|------|
| 実行頻度 | **日次**（毎朝06:00） |
| 計算時間 | AM便60秒 + PM便60秒 = 合計120秒 |
| データソース | CSV手動配置（`data/` フォルダ） |
| 出力 | ルートJSON + 配送指示書（テキスト） |
| 実行スクリプト | `scripts/run_pipeline.py` |

### cron 設定

```bash
# 毎朝6:00に当日の配送ルートを生成
0 6 * * * cd /path/to/delivery_routing && python scripts/run_pipeline.py
```

---

## 2. パイプライン構成

```
Step 1: データ取得（depot/customers/vehicles CSV）
  ↓
Step 2: 距離行列構築（Haversine × 道路係数1.3）
  ↓
Step 3: バリデーション（座標・容量・時間枠チェック）
  ↓
Step 4: AM/PM分割 + OR-Tools Routing最適化（各60秒）
  ↓  ※失敗時 → 最近傍法フォールバック
Step 5: 結果検証（全顧客カバー・時間枠・容量チェック）
  ↓
Step 6: ルートJSON + 配送指示書テキスト出力
```

### AM/PM分割ロジック

容量不足（需要815kg > 1便容量750kg）のため、2便制で運用:

| 便 | 出発時刻 | 分割基準 | 顧客数 | 需要 |
|-----|---------|---------|--------|------|
| AM便 | 08:00 | 時間枠中間値 ≤ 12:00 | 16件 | 635kg |
| PM便 | 12:30 | 時間枠中間値 > 12:00 | 4件 | 180kg |

各便を独立にOR-Tools Routing Solverで最適化。

## 3. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| 顧客データ空 | 0件 | **中断** |
| 車両データ空 | 0台 | **中断** |
| 需要 > 2便合計容量 | — | **中断**（2便でもカバー不可） |
| 単品需要 > 最大車両容量 | — | **中断** |
| 時間枠不正（end ≤ start） | — | **中断** |
| 座標が東京圏外 | — | 警告 |

---

## 4. 最適化手法

**OR-Tools Routing Solver**（solve_all.pyのam_pm_split相当）:

- 距離最小化（Haversine × 道路係数1.3）
- 容量制約: 各車両の積載量上限
- 時間枠制約: ソフト上限（遅延ペナルティ1000/件）
- 訪問義務: ドロップペナルティ100000/件（事実上ハード制約）
- 探索戦略: PATH_CHEAPEST_ARC + GUIDED_LOCAL_SEARCH

## 6. フォールバック手順

### パターン1: ソルバーが解なし

```
原因: 時間枠が厳しすぎる、顧客追加等
対応:
  1. パイプラインが自動で最近傍法フォールバックを実行
  2. 最近傍法も失敗 → exit code 1でパイプライン停止
  3. 手動対応: 前日のルートをベースに手動調整
```

### パターン2: 顧客の当日追加

```
対応: 最寄りの車両ルートに手動挿入
将来: パイプラインの再実行で自動対応
```

### パターン3: 車両故障（1台減）

```
対応: 2台 × 2便で再最適化
       V003（予備車両/小型200kg）を投入
```

### パターン4: 需要急増

```
対応: V003を投入 or 3便制に切り替え
検知: 総需要が1500kgを超えたら警告
```

## 8. 前提条件と注意事項

| 項目 | 値 | 備考 |
|------|-----|------|
| 移動速度 | 30 km/h | 渋滞は未考慮 |
| 道路係数 | 1.3 | 直線距離に対する実道路比 |
| デポ出発 | 08:00 / 12:30 | AM便 / PM便 |
| サービス時間 | 10-25分/件 | データ通り |

GPSログによる実走時間の取得が実現すれば、仮定の検証・精緻化が可能。
