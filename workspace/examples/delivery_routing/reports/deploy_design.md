# 配送ルート最適化 — 運用設計書

**作成日**: 2026-03-30
**対象**: 品川配送センター → 東京都内20顧客 × 3台（AM/PM分割運用）

---

## 1. 実行概要

| 項目 | 設定 |
|------|------|
| 実行頻度 | **日次**（毎朝6:00） |
| 計算時間 | AM便60秒 + PM便60秒 = 合計120秒 |
| データソース | CSV手動配置（`data/` フォルダ） |
| 出力 | 配送指示書（テキスト）+ ルートデータ（JSON） |
| 実行スクリプト | `scripts/run_daily.py` |

### cron 設定

```bash
0 6 * * * cd /path/to/delivery_routing && python scripts/run_daily.py
```

---

## 2. パイプライン構成

```
Step 1: データ取得（depot/customers/vehicles CSV）
  ↓
Step 2: バリデーション（座標・容量・時間枠チェック）
  ↓
Step 3: AM/PM分割（tw_end≤13:00→AM, tw_start≥11:00→PM）
  ↓
Step 4: OR-Tools Routing で各便を最適化（各60秒）
  ↓
Step 5: 結果検証（全顧客カバー・時間枠・容量チェック）
  ↓
Step 6: 配送指示書（txt）+ ルートデータ（JSON）出力
```

---

## 3. バリデーションルール

| チェック項目 | 閾値 | アクション |
|------------|------|----------|
| 顧客データ空 | 0件 | **中断** |
| 車両データ空 | 0台 | **中断** |
| 需要 > 2便容量 | — | **中断**（2便でもカバー不可） |
| 単品 > 最大車両容量 | — | **中断** |
| 座標が東京圏外 | — | 警告 |
| 時間枠不正（end≤start） | — | **中断** |

---

## 4. 監視指標

| 指標 | 正常範囲 | アラート条件 | 通知先 |
|------|---------|------------|--------|
| 配送完了率 | 20/20件 | 1件でも未配送 | 配車担当 |
| 時間枠違反 | 0件 | 1件以上 | 配車担当 |
| 容量超過 | 0件 | 1件以上 | 配車担当 |
| 総走行距離 | 100-120km | 150km超過 | 管理者 |
| 計算時間 | ~120秒 | 300秒超過 | システム担当 |

---

## 5. フォールバック

| パターン | 対応 |
|---------|------|
| ソルバーが解なし | 前日のルートをベースに手動調整 |
| 顧客追加（当日） | 最寄りの車両ルートに手動挿入 |
| 車両故障（1台減） | 2台×2便で再最適化（V003を投入） |
| 需要急増 | V003を投入 or 3便制に切り替え |

---

## 6. モデル見直しトリガー
