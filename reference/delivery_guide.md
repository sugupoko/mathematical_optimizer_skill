# 最適化結果の届け方ガイド（delivery_guide.md）

**最適化は「解けた」がゴールではない。現場に使ってもらって初めて価値が出る。**

このガイドは、ヒアリング結果から適切な納品形態を選び、現場に定着させるための判断基準をまとめたもの。

---

## 1. なぜ届け方が重要か

最適化プロジェクトの失敗パターンの大半は「解けない」ではなく「使ってもらえない」。

| 失敗パターン | 原因 | 対策 |
|-------------|------|------|
| ブラックボックス問題 | なぜその解か説明できない → 信用されない | 制約チェック・スコア根拠の可視化 |
| 現場不在の設計 | 使う人が設計に関与していない | ヒアリングで運用フローを押さえる |
| 神 Excel に戻る | 最適化ツールより Excel の方が使いやすい | 既存フォーマットを踏襲する |
| 手修正できない | 急な変更に対応できず運用が破綻 | 手動上書き + 制約チェック機能 |
| 過複雑なUI | 機能を盛りすぎて学習コストが高い | まず最小限で出して段階的に拡張 |

**最重要原則: 最適化結果は「最適解」ではなく「修正可能な草案」として提示する。**

## 2. 判断軸 — ヒアリングで押さえる5つの観点

### 観点①: 利用者のリテラシー

| レベル | 特徴 | 向いている形式 |
|--------|------|--------------|
| PCを使わない | 現場作業者、ドライバー | 紙 / PDF / スマホで見るだけ |
| Excel は使える | 事務職、チームリーダー | Excel 書き戻し |
| ブラウザは使える | 一般オフィスワーカー | Web UI（Streamlit 等） |
| IT に詳しい | 情シス、データ分析者 | API / Jupyter / 自由度の高いツール |

**ヒアリングの聞き方:**
- 「普段どんなツールで仕事してますか？」
- 「Excel のマクロは使いますか？」
- 「社内で Web アプリを使う文化はありますか？」

### 観点②: 更新頻度と変更への対応

| 頻度 | 特徴 | 向いている形式 |
|------|------|--------------|
| 月1回 | 作成→配布で完結 | Excel / PDF 手渡し |
| 週1回 | 定例の作成作業 | Excel + スクリプト自動生成 |
| 毎日 | 自動化が必須 | Web + 自動生成パイプライン |
| リアルタイム | 当日変更への即応 | Web + 再最適化機能 |

**ヒアリングの聞き方:**
- 「急な変更（欠勤、急ぎの注文）はどのくらいの頻度で起きますか？」
- 「変更が起きたとき、今はどう対応していますか？」
- 「何分以内に対応する必要がありますか？」

### 観点③: 結果の確認レベル

| レベル | ユーザーの期待 | 必要な機能 |
|--------|-------------|----------|
| **見るだけ** | 結果を受け取って使う | PDF / 印刷 / メール配信 |
| **納得したい** | なぜこの配置か理解したい | 制約チェックシート、スコア比較、KPI表示 |
| **触りたい** | 自分で一部を変えたい | 手修正 + 制約チェック |
| **試したい** | 「こうしたらどうなる？」を試したい | What-if シミュレーション、シナリオ比較 |

**ヒアリングの聞き方:**
- 「ソルバーが出した結果をそのまま使いますか？」
- 「『なぜこの人がこのシフトに？』と聞かれた時に説明が必要ですか？」
- 「『人を1人増やしたらどうなる？』のようなシミュレーションをしたいですか？」

### 観点④: 利用人数と共同作業

| パターン | 特徴 | 向いている形式 |
|---------|------|--------------|
| 1人で作成→配布 | 管理者が作って配る | Excel（ローカル） |
| 数人で確認・承認 | 作成→確認→修正→確定 | Excel + メール / Google Sheets |
| チーム全員で閲覧 | 各自が自分の分を確認 | Web（閲覧用）/ PDF 個人配布 |
| 複数人で同時編集 | リアルタイム共同作業 | Google Sheets / Web アプリ |

**ヒアリングの聞き方:**
- 「結果を見る人は何人くらいですか？」
- 「確認・承認のフローはありますか？」
- 「同時に編集する必要がありますか？」

### 観点⑤: デプロイ環境と制約

| 環境 | 向いている形式 |
|------|--------------|
| オフライン・閉域網 | Excel / デスクトップアプリ |
| 社内サーバーあり | Streamlit / Retool（セルフホスト） |
| クラウド利用可 | Google Sheets / SaaS |
| スマホのみ | モバイル対応 Web / LINE 連携 |

**ヒアリングの聞き方:**
- 「社外のクラウドサービスは使えますか？」
- 「社内にサーバーを立てられますか？」
- 「現場ではPCとスマホのどちらを使いますか？」

## 3. 納品形態の選択肢

### Excel 書き戻し

```
向いている: リテラシー中、更新頻度低〜中、1人作成、オフライン可
メリット: 学習コストゼロ、現場のフォーマットを踏襲できる
デメリット: 手修正時のリアルタイム制約チェックが難しい
実装: openpyxl（Python）で色分け・サマリ付き Excel を自動生成
拡張: VBA マクロで制約チェックボタンを追加可能
```

### Google Sheets + Apps Script

```
向いている: 複数人で確認・編集、クラウド利用可
メリット: 共同編集、バージョン履歴、Apps Script で制約チェック自動化
デメリット: 閉域網で使えない、大規模データに弱い
実装: gspread（Python）で結果を書き込み、Apps Script で制約チェック
```

### PDF / 印刷

```
向いている: 見るだけ、PC を使わない現場、配布のみ
メリット: 誰でも見れる、改ざんされない
デメリット: 修正不可
実装: matplotlib / reportlab で生成
```

### Streamlit Web アプリ

```
向いている: 手修正あり、What-if したい、社内サーバーあり
メリット: Python だけで完結、ドラッグ&ドロップも可能、リアルタイム制約チェック
デメリット: 大規模同時接続に弱い、デザインの自由度は限定的
実装: st.dataframe で表編集、st.map で地図、st.slider でパラメータ操作
具体例: 手修正 → 再最適化ボタン → 制約チェック → 結果比較
```

### Bryntum / DHTMLX（商用ガントチャート）

```
向いている: ドラッグ&ドロップ必須、サービスとして売る、大規模
メリット: プロフェッショナルなUI、ドラッグ操作が直感的
デメリット: ライセンス費用あり、JavaScript 開発が必要
事例: Timefold（最適化エンジン）+ Bryntum（UI）の公式連携が確立
```

### Retool / Budibase（ローコード内部ツール）

```
向いている: 複数画面が必要、DB 連携あり、社内ツールとして継続運用
メリット: DB/API直結、権限管理、ワークフロー内蔵
デメリット: 商用（Retool）/ セルフホスト（Budibase）
実装: 最適化 API をバックエンドに、フロントをローコードで構築
```

### Power BI / Tableau

```
向いている: 経営層向けダッシュボード、既存BI環境に乗せたい
メリット: 既に導入済みなら追加コストなし、見栄えが良い
デメリット: インタラクティブな再最適化は困難
実装: 最適化結果の CSV/DB を接続、KPI ダッシュボードとして表示
```

## 4. 問題タイプ別の可視化推奨

| 問題タイプ | 見るだけ | 触りたい | 試したい |
|-----------|---------|---------|---------|
| シフト最適化 | 色分き表（Excel/PDF） | ガントチャート（drag&drop） | What-if（人数変更シミュレーション） |
| 配送ルート | ルート地図（Leaflet/PDF） | 地図上でポイント移動 | 車両台数変更シミュレーション |
| マッチング | 対応表（Excel） | 対応線の付替え | 条件変更時の再マッチング |
| チケットアサイン | カンバンボード | カード移動（drag&drop） | 優先度変更シミュレーション |
| 施設配置 | 地図上にピン | ピン移動で再計算 | 候補地追加のシミュレーション |

### 可視化ライブラリの選択

| 用途 | 静的（見るだけ） | インタラクティブ |
|------|---------------|----------------|
| 表・ガントチャート | matplotlib / openpyxl | Streamlit / Bryntum / DHTMLX |
| 地図 | matplotlib + cartopy | Leaflet.js / Kepler.gl / folium |
| グラフ（棒・折線） | matplotlib | Plotly / Altair |
| ネットワーク図 | networkx + matplotlib | vis.js / D3.js |

## 5. 現場定着のための設計原則

### 原則1: 既存フォーマットを踏襲する

現場が今使っている Excel やホワイトボードの形式に合わせる。
「新しいツールを覚えてください」は最大の導入障壁。

### 原則2: 結果は「草案」として出す

最適化結果を押し付けない。「提案です。ここを変えたければ変えてください」というスタンス。
手動上書き機能は必須。

### 原則3: 「なぜこの配置か」を説明する

制約チェックシート、スコアの内訳、「この人をここに入れた理由」の表示。
ブラックボックスは信用されない。

### 原則4: KPI は現場の言葉で出す

- ✗ 「目的関数値: 1234」
- ○ 「残業削減: 1日あたり平均 2.3 時間」
- ○ 「未充足シフト: 2件（火・金の夜勤）」
- ○ 「公平性: 最多40h、最少24h、差16h」

### 原則5: 小さく始めて段階的に拡張する

```
Phase 1: Excel 書き戻し（現行踏襲、学習コストゼロ）
Phase 2: 制約チェック機能の追加（VBA or スクリプト）
Phase 3: Web 化 or ドラッグ&ドロップ（手修正が多い場合のみ）
Phase 4: What-if / シナリオ比較（経営判断に使う場合）
```

全社一斉展開より、1チームで成功体験を作る方が効果的。
研究では、構造化されたチェンジマネジメントで採用率 +60%、導入期間 -45%。

## 6. ヒアリング結果 → 納品形態の判断フロー

```
手修正が必要？
├── No → 見るだけ
│     ├── PC使える → Excel / Web ダッシュボード
│     └── PC使わない → PDF / 印刷
│
└── Yes → どのレベル？
      ├── セル書き換え程度 → Excel（現行踏襲）
      ├── ドラッグ&ドロップで直感的に → Web UI（Streamlit / Bryntum）
      └── 変えた後に再最適化したい → Streamlit + ソルバー連携
```

```
What-if が必要？
├── No → 上記で決定
└── Yes → Streamlit（スライダー + 再計算）or Power BI（シナリオ比較）
```

```
複数人で同時編集？
├── No → Excel / Streamlit
└── Yes → Google Sheets / Web アプリ（DB 連携）
```
