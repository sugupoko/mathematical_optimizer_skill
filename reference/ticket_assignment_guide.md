# チケットアサイン最適化ガイド

ITSMチケット（インシデント、サービスリクエスト、バグ等）を
エンジニアに最適に割り当てる問題の考え方・手法選択・評価方法をまとめたガイド。
`ticket_assignment_template.py` と合わせて使う。

---

## チケットアサインが必要な場面

「どのチケットを誰に割り当てるか」を決める問題は、IT運用のあらゆる場面に存在する。

| 分野 | チケットの例 | 割当先 | 典型的な制約 |
|------|-------------|--------|-------------|
| **ITSMインシデント管理** | サーバ障害、ネットワーク断 | L1/L2/L3エンジニア | スキル、ティア、SLA、ブロック状態 |
| **サービスリクエスト処理** | アカウント発行、権限変更 | オペレータ | スキル、優先度、処理時間 |
| **バグトラッキング（Jira等）** | バグ修正、機能改善 | 開発者 | スキル、スプリント容量、依存関係 |
| **カスタマーサポート** | 問い合わせ、クレーム | サポート担当 | 言語、製品知識、VIP対応 |
| **ITIL変更管理** | 変更要求（RFC） | 変更実施者 | 承認ステータス、実施ウィンドウ、スキル |

共通点: **スキルマッチ + 負荷バランス + 緊急度**の3軸で割当を決める。
シフト最適化やマッチング問題とは異なり、**動的に状況が変わる**点が特徴。

---

## 問題の分類

問題の特徴を見て、手法を選ぶ。

| 特徴 | 推奨手法 | 理由 |
|------|---------|------|
| 静的（1回の一括割当） | **CP-SAT / ハンガリアン法** | 全体最適が得られる |
| 動的（15分おきに再最適化） | **CP-SAT + 差分計算** | 状況変化に追従しつつ最適性を維持 |
| リアルタイム（即時振り分け） | **貪欲法 + 定期バッチ再最適化** | 即応性と最適性のバランス |

**迷ったらCP-SAT + 定期バッチ**。ほとんどのITSM環境では15分間隔のバッチ再最適化で十分。

## シフト最適化との違い

チケットアサインは一見シフト最適化に似ているが、本質的に異なる点が多い。
混同するとモデリングを誤るので注意。

| 比較軸 | シフト最適化 | チケットアサイン |
|--------|-------------|----------------|
| **決定変数** | 誰をいつどのシフトに入れるか | どのチケットを誰に割り当てるか |
| **時間軸** | 1週間〜1ヶ月先を計画 | 今あるチケットを今割り当てる |
| **制約の性質** | 労働法、連勤制限など静的ルール | スキル、ティア、SLA、ブロック状態 |
| **動的性** | 計画時に全情報が揃う | チケットは随時到着する |
| **ブロック状態** | なし（シフトに「ブロック」はない） | あり（外部待ちで進捗が止まる） |
| **再割当** | ほぼなし（シフト交代は例外処理） | 頻繁（滞留検知→再アサイン） |
| **目的関数** | 公平性・希望充足が中心 | SLA遵守・スキル適合が中心 |

**重要な違い**: シフト最適化は「未来を計画する」問題だが、
チケットアサインは「今の状況に対応する」問題。
そのため、チケットアサインでは**状態管理**（誰が何件持っているか、
どのチケットが滞留しているか）が決定的に重要になる。

## ティア階層制約

### L1 / L2 / L3 の構造

ITSMでは、エンジニアをスキルレベル（ティア）で階層化するのが一般的。

```
L3（専門家）    ← 最も高スキル。エスカレーション先。人数少。
  ↑ エスカレーション
L2（中堅）      ← 大半のインシデントを処理。主力。
  ↑ エスカレーション
L1（一次対応）  ← 定型的な対応。ヘルプデスク。人数多。
```

| ティア | 対応範囲 | 典型的な人数比 |
|--------|---------|---------------|
| L1 | 定型作業、ナレッジベースで解決可能な問い合わせ | 多い（60%） |
| L2 | 調査が必要なインシデント、設定変更を伴う作業 | 中程度（30%） |
| L3 | 根本原因分析、アーキテクチャ変更、ベンダー連携 | 少ない（10%） |

### CP-SATでのモデリング

```python
# ティア階層: 上位ティアは下位ティアの作業も可能
tier_rank = {"L1": 1, "L2": 2, "L3": 3}

# ハード制約: エンジニアのティア >= チケットの必要ティア
for t_idx, ticket in enumerate(tickets):
    required_rank = tier_rank[ticket["required_tier"]]
    for e_idx, eng in enumerate(engineers):
        eng_rank = tier_rank[eng["tier"]]
        if eng_rank < required_rank:
            model.add(x[t_idx, e_idx] == 0)  # 割当禁止
```

**注意**: L3エンジニアにL1チケットを割り当てるのは「可能」だが「非効率」。
目的関数でティア差が大きい割当にペナルティを与えるのが良い実践。

```python
# ソフト制約: ティアのオーバースペック割当にペナルティ
tier_gap = eng_rank - required_rank  # 0なら適正、大きいほど過剰
penalty = tier_gap * 5  # 1段階超過ごとに-5点
```

## ブロック状態の扱い

### ブロックとは

チケットの進捗が**外部要因**で止まっている状態。
担当エンジニアの責任ではなく、待ちの間は作業できない。

| ブロック種別 | 内容 | 典型的な待ち時間 |
|-------------|------|----------------|
| **vendor** | ベンダーからの回答待ち | 数時間〜数日 |
| **customer** | 顧客からの情報提供待ち | 数時間〜1週間 |
| **approval** | 上長・変更管理委員会の承認待ち | 数時間〜数日 |
| **parts** | 部品・ライセンスの調達待ち | 数日〜数週間 |
| **maintenance** | メンテナンスウィンドウ待ち | 数日（次の深夜帯まで） |

### スロット消費ポリシー

ブロック中チケットがエンジニアのスロットを「消費する」かどうかは、
組織のポリシーにより2つの考え方がある。

**ポリシーA: スロットを消費する（デフォルト）**

```python
# ブロック中でもスロットを消費する
# → エンジニアの実質的なキャパシティが減る
# → 保守的（エンジニアに余裕を持たせる）
eng_states = build_engineer_state(engineers, tickets, release_blocked=False)
```

- メリット: ブロック解除時にすぐ対応に戻れる余裕がある
- デメリット: ブロックが多いとスロットが足りなくなる

**ポリシーB: スロットを解放する**

```python
# ブロック中チケットはスロットを消費しない
# → エンジニアは他のチケットに取りかかれる
# → 攻撃的（スループット重視）
eng_states = build_engineer_state(engineers, tickets, release_blocked=True)
```

- メリット: スループットが上がる
- デメリット: ブロック解除時にエンジニアの手が空いていない可能性

**実務的な指針**: ブロック解除の頻度が1日1回程度なら `release_blocked=True` が効率的。
頻繁に解除される（数十分単位）なら `release_blocked=False` が安全。

## 滞留検知

### 期待進捗の計算

滞留検知の核心は「期限に対してどのくらい進んでいるべきか」の期待進捗を計算し、
実際の進捗との乖離を測ること。

```
期待進捗 (%) = (経過時間 / 期待解決時間) × 100

乖離 (pt) = 期待進捗 - 実際進捗
```

期待解決時間は以下の優先順位で決まる:
1. LLM推定の残り時間 + 経過時間
2. 優先度別のSLAデフォルト（P1=4h, P2=8h, P3=24h, P4=72h）

### 閾値のチューニング

| レベル | 乖離（pt） | 意味 | アクション |
|--------|-----------|------|-----------|
| **normal** | < 25 | 正常範囲 | 何もしない |
| **warning** | 25 - 49 | やや遅延 | リーダーに通知 |
| **critical** | ≥ 50 | 大幅遅延 | 再アサインを検討 |
| **blocked** | — | 外部要因停止 | ブロック解除を追跡 |

**閾値は組織に合わせて調整すること**。
厳格なSLAがある環境では warning を 15pt に下げるなど、
過去のSLA違反率を見ながら最適な閾値を見つける。

### 滞留とブロックの区別

これは**決定的に重要な区別**。

| | 滞留（stagnation） | ブロック（blocked） |
|---|---|---|
| 原因 | 担当者の手が回っていない、スキル不足 | 外部要因（ベンダー、顧客、承認待ち） |
| 対策 | **再アサイン** | **ブロック解除の追跡** |
| スロット影響 | スロットを消費し続ける | ポリシーにより解放可能 |
| 責任 | 担当者（または割当の問題） | 組織プロセスの問題 |

ブロック中のチケットを「滞留」と誤判定して再アサインしても意味がない。
誰が持っても進まないのだから。

## LLM推定の活用

### 推定のフォールバックチェーン

解決時間の推定精度は、利用可能なデータ量に依存する。
データが豊富な場合ほど信頼度が高く、データがない場合はデフォルト値に頼る。

```
推定リクエスト
  │
  ├─ 個人履歴（スキル×エンジニア）≥ 3件あるか？
  │   YES → 中央値ベース推定（信頼度 0.7〜0.9）
  │   NO ↓
  │
  ├─ 個人履歴 1〜2件あるか？
  │   YES → 平均値ベース推定（信頼度 0.4）
  │   NO ↓
  │
  ├─ カテゴリ（スキル）の履歴があるか？
  │   YES → カテゴリ中央値（信頼度 0.3）
  │   NO ↓
  │
  └─ 優先度別デフォルト（信頼度 0.1）
      P1=4h, P2=8h, P3=24h, P4=72h
```

### 信頼度レベルの使い方

| 信頼度 | 意味 | 目的関数での扱い |
|--------|------|----------------|
| 0.7〜0.9 | 十分な個人履歴に基づく推定 | ボーナススコアを付与（推定が信頼できる割当を優先） |
| 0.4 | 少数の個人履歴 | 小さなボーナス |
| 0.3 | カテゴリ平均 | ほぼニュートラル |
| 0.1 | デフォルト値 | ペナルティなし（情報がないだけ） |

**注意**: 信頼度が高い = 良い割当、ではない。
信頼度が高いということは「この人が解決するのにかかる時間の予測が正確」という意味であり、
その推定時間が短いかどうかは別の問題。

### 精度改善サイクル

推定精度は時間とともに改善できる。

```
1. 初期: デフォルト値のみ（信頼度 0.1）
     ↓ チケット対応が蓄積される
2. 1ヶ月後: カテゴリ平均が使える（信頼度 0.3）
     ↓ 個人の対応履歴が蓄積される
3. 3ヶ月後: 個人履歴が使える（信頼度 0.7+）
     ↓ 推定と実績の乖離を分析
4. 継続的: 外れ値の除去、経験年数補正の係数チューニング
```

経験年数による補正（ベテランほど速い）:

```python
# 経験0年 → 補正なし（×1.0）
# 経験5年 → 15%短縮（×0.85）
# 経験10年以上 → 30%短縮（×0.7）で頭打ち
factor = max(0.7, 1.0 - 0.03 * min(experience_years, 10))
adjusted_hours = base_hours * factor
```

## 動的再最適化のパターン

チケットは常に到着し、状況は常に変わる。
1回解いて終わりではなく、定期的に再最適化する必要がある。

### パターン1: バッチ再最適化（定期実行）

```
[毎15分] 未割当チケット + 現在の状態 → CP-SAT → 新規割当
```

- **頻度**: 5〜30分間隔（環境に応じて調整）
- **メリット**: 実装がシンプル。全体最適が得られる。
- **デメリット**: チケット到着から最大15分の遅延。
- **適するケース**: ほとんどのITSM環境。P1の即時対応が不要な場合。

### パターン2: イベント駆動（P1到着時に即時実行）

```
[P1到着] → 即時でCP-SAT実行（P1だけ or 全体）
[P2-P4] → 通常のバッチサイクルで処理
```

- **メリット**: P1の対応開始が速い。
- **デメリット**: P1が連続すると再最適化が頻発する。
- **適するケース**: SLAが厳しくP1の初動時間が重要な環境。

### パターン3: 差分再最適化（変更分のみ）

```
[毎15分] 前回からの差分（新規チケット + 状態変化のみ）→ CP-SAT
既存の割当はそのまま維持。新規分だけ最適化。
```

- **メリット**: 既存割当が安定する（担当者が頻繁に変わらない）。
- **デメリット**: 全体最適からは遠ざかる。
- **適するケース**: 再アサインのコスト（引き継ぎ時間等）が大きい環境。

### パターン4: 再アサインあり

```
[毎15分]
  1. 滞留検知（全割当済みチケット）
  2. critical な滞留 → 再アサイン候補
  3. 新規チケット + 再アサイン候補 → CP-SAT
```

- **メリット**: 滞留を自動検知して解消できる。
- **デメリット**: 引き継ぎコストが発生する。
- **適するケース**: SLA違反の削減が最優先の環境。

### 再アサインコストの考慮

再アサインには隠れたコスト（引き継ぎ時間、コンテキスト切替）がある。
これを目的関数に組み込む。

```python
# 再アサインボーナス: 滞留解消の価値
reassign_bonus = 25  # 再アサイン候補のチケットには+25点

# 引き継ぎペナルティ（ここでは暗黙的に考慮）
# → 再アサイン先のエンジニアは追加の学習コストを払う
# → 推定解決時間に引き継ぎ分を加算するのが正確
```

## 判断フロー

チケットアサイン問題に出会ったときの手法選択フロー。

```
チケットアサイン問題に出会った
  │
  ├── チケット数は？
  │     ├── 小規模（〜50件/日） → 手動割当 + ルールベースでも十分
  │     ├── 中規模（50〜500件/日） → CP-SAT バッチ最適化
  │     └── 大規模（500件+/日） → 貪欲法（即時）+ CP-SAT（バッチ補正）
  │
  ├── 割当の頻度は？
  │     ├── 1回の一括割当（バッチ） → CP-SAT
  │     ├── 定期再最適化（N分おき） → CP-SAT + cron
  │     ├── リアルタイム（即時） → 貪欲法（即時）+ CP-SAT（バッチ補正）
  │     └── イベント駆動（P1到着時） → CP-SAT（P1トリガー）+ バッチ（通常）
  │
  ├── 再アサインは必要か？
  │     ├── 不要（一度割り当てたら変えない） → 通常のCP-SAT
  │     └── 必要（滞留時に別の人へ） → 滞留検知 + 再アサインコスト考慮
  │
  └── ブロック状態はあるか？
        ├── なし → 標準モデルで対応
        └── あり → スロット消費ポリシーを決めてモデルに組み込む
```

## 評価指標

最適化の効果を測るために、以下の指標を追跡する。

### 主要指標

| 指標 | 計算方法 | 目標 | 意味 |
|------|---------|------|------|
| **SLA遵守率** | SLA内に解決したチケット / 全チケット | ≥ 95% | 最も重要なKPI |
| **割当率** | 割当成功数 / 割当対象数 | ≥ 90% | 割り当てられないチケットが残っていないか |
| **スキル一致率** | スキルマッチ割当 / 全割当 | ≥ 85% | 適切な人に割り当てているか |

### 補助指標

| 指標 | 計算方法 | 意味 |
|------|---------|------|
| **負荷の公平性（標準偏差）** | 各エンジニアの負荷率の標準偏差 | 小さいほど公平 |
| **再アサイン率** | 再アサインされたチケット / 全チケット | 高すぎると引き継ぎコストが問題 |
| **平均解決時間** | 解決までの時間の平均 | 短いほど良い |
| **初回割当適正率** | 再アサインなしで解決 / 全チケット | 高いほど初回の割当精度が良い |
| **ティア適正率** | 適正ティアへの割当 / 全割当 | L3にL1作業をさせていないか |
| **HC違反数** | ハード制約違反の件数 | 0が必須 |

### 評価コード例

```python
from ticket_assignment_template import (
    LLMEstimator, build_engineer_state,
    solve_ticket_assignment, evaluate_assignment,
)

# ... データ準備 ...

assignments, info = solve_ticket_assignment(tickets, eng_states, estimator, tiers)
metrics = evaluate_assignment(assignments, tickets, eng_states, estimator)

print(f"割当率: {metrics['rate']*100:.0f}%")
print(f"スキル一致率: {metrics['skill_match']*100:.0f}%")
print(f"HC違反: {metrics['hc_violations']}件")
print(f"負荷公平性（σ）: {metrics['load_std']:.2f}")
```

## よくある落とし穴

### 1. スキルマッチだけで割り当てる

スキルが合う人の中で最も空いている人に割り当てる — 一見合理的だが、
SLA緊急度を考慮しないとP1チケットがL1の新人に回ってしまう。

**対策**: スキルマッチは必要条件（ハード制約）、
SLA緊急度とスキルレベルは目的関数でバランスを取る。

### 2. L3エンジニアを使い尽くす

L3は人数が少なく替えが利かない。L1/L2で対応可能なチケットまでL3に回すと、
本当にL3が必要なチケットが来たときに対応できない。

**対策**: ティアのオーバースペック割当にペナルティを与える。
L3のスロットは「バッファ」として一定数空けておく。

### 3. ブロック状態を無視する

ブロック中チケットを考慮しないと、エンジニアが「見かけ上は空いている」のに
実際は5件のブロック解除待ちを抱えている — ブロックが解除された瞬間に破綻する。

**対策**: `build_engineer_state()` でブロック中チケットを明示的に管理する。

### 4. 再アサインのコストを軽視する

再アサイン = 引き継ぎ = 新しい担当者がゼロから調査し直す。
元の担当者の調査時間が無駄になる。

**対策**: 再アサインは `critical` レベルの滞留のみに限定する。
`warning` で再アサインすると引き継ぎコストが改善効果を上回ることが多い。

### 5. 公平性を無視する

効率だけ追求すると、ベテランにチケットが集中し、新人が成長できない。
長期的にはチームの対応力が低下する。

**対策**: 目的関数に公平性ペナルティ（負荷率の偏りを抑制）を組み込む。
負荷標準偏差を評価指標として追跡する。
