# 最適化スペシャリストの思考回路 — 抽象化と判断基準

## この文書の目的

最適化スペシャリストが問題に取り組むときの思考パターンを言語化し、「この状況ではこう考える」という判断基準を紐づける。LLMや非専門家が最適化問題に取り組む際のガイドとして使う。

---

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  ① 問題を小さくできないか？（分解・縮約）            │
│         ↓                                           │
│  ② 何が一番きついか？（ボトルネック特定）            │
│         ↓                                           │
│  ③ 固定できる部分はないか？（自由度の削減）          │
│         ↓                                           │
│  ④ この問題は何かに似ているか？（パターン認識）      │
│         ↓                                           │
│  ⑤ どう変えたら良くなるか？（ムーブ設計）            │
│         ↓                                           │
│  ⑥ どこまで良くなり得るか？（下界・限界の把握）      │
│         ↓                                           │
│  ⑦ 何が不可能か？（不可能性の証明）                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## ① 問題を小さくできないか？（分解・縮約）

### 思考

> 「この問題をそのまま解こうとするな。分けられないか？固定できないか？次元を減らせないか？」

大きな問題を小さな部分問題に分解すると、各部分が解きやすくなる。最適化スペシャリストが最初に考えることは「全体を一度に解く」ではなく「どう分ければ各部分が簡単になるか」。

### 分解の方法と選び方

| 分解方法 | いつ使うか | 実例 |
|---------|----------|------|
| **時間で分割** | 時間軸がある問題で、時間帯ごとに独立性が高い | シフト最適化のAM/PM分割、配送ルートの午前/午後分離 |
| **空間で分割** | 地理的な広がりがある問題 | 配送ルートのクラスタ分割、エリアごとのルート計画 |
| **主体で分割** | 人や車両ごとに独立に計画できる | 作業者ごとの勤務パターン生成→組合せ選択（列生成法） |
| **段階で分割** | 決定を順序立てられる | まず割当→次に順序、まず制約充足→次に品質改善 |
| **制約で分割** | 制約の一部を緩和して解き、後で戻す | 反復緩和（expG14）、ラグランジュ緩和 |
| **逐次構築** | N個の解がN+1個に拡張できる | Santa 2025のL字テクニック、段階的な制約追加 |

### 分解してはいけない方向

> 分解は制約の構造に沿って行う。構造を無視した分解は逆効果。

| 良い分解 | 悪い分解 |
|---|---|
| AM/PM分割（時間制約に沿っている） | ランダムに半分に分割 |
| 地理クラスタ（距離制約に沿っている） | アルファベット順に分割 |
| 時間帯で分割（休憩制約を自然に尊重） | 「いつ働くか」と「何をするか」を分離（スキル制約を破壊） |

### 判断フロー

```
問題の規模は？
  ├── 小さい（~100変数）→ 分解不要、ソルバーで直接解く
  └── 大きい（1000+変数）→ 分解が必要
        │
        問題に時間軸があるか？
        ├── ある → 時間で分割を検討
        └── ない → 空間・主体で分割を検討
              │
              分割後の部分問題は独立か？
              ├── ほぼ独立 → 分割して別々に解く
              └── 境界で依存 → オーバーラップ付き分割（Rolling Horizon）
```

## ② 何が一番きついか？（ボトルネック特定）

### 思考

> 「全部の制約を均等に見るな。一番きつい制約が解の品質を支配している。そこを攻めろ。」

### ボトルネック特定の方法

| 手法 | やり方 | わかること |
|------|--------|----------|
| **余裕度分析** | 各制約の（供給÷需要）を計算 | 余裕度が最小の制約がボトルネック |
| **制約緩和分析** | 制約を1つずつ外して解く | 外したとき最も改善する制約がボトルネック |
| **違反パターン分析** | ベースラインの違反を集計 | 最も多く違反する制約がボトルネック |
| **感度分析** | 制約のパラメータを少し変えて解く | 少しの変更で大きく結果が変わるパラメータがボトルネック |

### 本プロジェクトでの実例

| プロジェクト | ボトルネック | 特定方法 | 対策 |
|---|---|---|---|
| シフト最適化 | HC9（訓練ペアリング） | 違反パターン分析 | Trainer-first構築 |
| シフト最適化 | HC3+HC9の相互作用 | 900回全敗で証明 | グローバル推論が必要 |
| 配送ルート | 時間制約（2時間以内の納品） | 全31違反がこの制約 | AM/PM分割 |
| Santa 2025 | 衝突判定の計算コスト | プロファイリング | GPU加速、sparrow |

### 判断フロー

```
ベースラインを実行
  ↓
違反はあるか？
  ├── ない → ソフト制約のスコアが低い部分を分析
  └── ある → 違反の内訳を見る
        │
        1種類の制約に集中しているか？
        ├── 集中 → その制約がボトルネック → 集中対策
        └── 分散 → 制約緩和分析で1つずつ外して最も効く制約を特定
```

## ③ 固定できる部分はないか？（自由度の削減）

### 思考

> 「全部を同時に最適化しようとするな。動かさなくていい部分を固定して、本当に動かすべき部分だけを最適化しろ。」

探索空間が大きいほど良い解を見つけにくい。固定できる変数を固定して探索空間を小さくするのが定石。

### 固定の方法と選び方

| 固定方法 | いつ使うか | 実例 |
|---------|----------|------|
| **良い解の大部分を保存** | 既に良い解がある場合 | ヒューリスティック解の69%を保存してソルバーで修復（expG01） |
| **明らかな割当を先に確定** | 選択肢が1つしかない変数 | 「この工程ができるのは彼だけ」→ 先に確定 |
| **逐次確定** | 前の決定が後の選択肢を絞る | N本のパッキングを固定してN+1本目だけ探す（Santa 2025 L字） |
| **次元を落とす** | 連続変数の一部を固定 | 「正方形の高さを固定して幅だけ最適化」（Santa 2025 2位） |

### 判断フロー

```
現在の解はあるか？
  ├── ある（品質は良いが制約違反）
  │     → 違反していない部分を固定、違反部分だけ再最適化
  └── ない
        │
        確実に決まる変数はあるか？
        ├── ある → 先に確定して探索空間を縮小
        └── ない → 段階的に確定していく（逐次構築）
```

## ④ この問題は何かに似ているか？（パターン認識）

### 思考

> 「車輪の再発明をするな。この問題は既知の問題クラスに帰着できないか？帰着できれば、その分野の50年分の研究成果が使える。」

### よくある帰着パターン

| 問題の特徴 | 帰着先 | 使えるツール |
|---|---|---|
| 「誰をいつどこに割り当てるか」 | **割当問題/スケジューリング** | CP-SAT、MIP |
| 「どの順番で回るか」 | **巡回セールスマン問題(TSP)** | LKH、OR-Tools Routing |
| 「容量制限付きで回る」 | **容量制約付きVRP (CVRP)** | HGS、OR-Tools |
| 「時間枠内に回る」 | **時間枠付きVRP (VRPTW)** | Solomon型ソルバー |
| 「箱に詰める」 | **ビンパッキング** | First Fit Decreasing |
| 「2グループに分ける」 | **グラフ分割** | METIS、スペクトラル法 |
| 「並べ替える」 | **TSP変種** | k-opt、LKH |
| 「組合せを選ぶ」 | **集合被覆/ナップサック** | MIP、貪欲法 |

### 意外な帰着の例

| 元の問題 | 一見すると | 実は |
|---|---|---|
| Santa 2024（文章並べ替え） | 自然言語処理 | **TSP**（k-opt、double-bridgeが効いた） |
| Santa 2021（映画スケジュール） | スケジューリング | **最短共通超文字列 → TSP** |
| 時間制約付き配送 | 普通のVRP | **Open VRPの変種**（帰りの時間が重要） |

### 判断フロー

```
問題の決定変数は何か？
  ├── 割当（誰を何に）→ 割当問題/スケジューリング
  ├── 順序（何を先に）→ TSP / スケジューリング
  ├── 経路（どう回る）→ VRP
  ├── 配置（どこに置く）→ パッキング
  └── 選択（何を選ぶ）→ ナップサック / 集合被覆
        ↓
帰着先の既存ソルバーを調べる
  ├── OSS（OR-Tools, LKH, HGS等）→ まず動かす
  └── 商用（Gurobi, CPLEX）→ 品質が必要なら検討
```

## ⑤ どう変えたら良くなるか？（ムーブ設計）

### 思考

> 「解を少し変えてもっと良くする。この『少し変える』方法の設計が最適化の腕の見せ所。問題に合ったムーブを自分で発明しろ。」

### ムーブの種類と選び方

| ムーブの種類 | いつ使うか | 例 |
|---|---|---|
| **汎用ムーブ** | まず試す。ベースラインとして | swap, 2-opt, relocate, insert, remove |
| **問題特化ムーブ** | 汎用が効かない時に設計 | 重量考慮swap（Santa 2015）、trainer-first配置（シフト） |
| **破壊的ムーブ** | 局所最適から脱出したい時 | LNSの30%破壊、クラスタ再編 |
| **修復ムーブ** | 制約違反を直したい時 | ejection chain、IFSバックトラック |

### 良いムーブの3条件

1. **差分計算が高速** — ムーブ後のスコアをO(1)で計算できる
2. **到達性がある** — 任意の解から最適解に到達できるムーブ系列が存在する
3. **問題の構造を活かす** — 「近いものを入れ替える」「きつい制約を緩和する方向」

### ムーブを組み合わせる（メタヒューリスティクスの選択）

| メタヒューリスティクス | 特徴 | いつ選ぶか |
|---|---|---|
| **焼きなまし法（SA）** | 悪化も確率的に受理。最も汎用的 | 迷ったらまずこれ |
| **タブー探索** | 循環を防ぐ。戦略的振動 | feasibility境界を攻めたい時 |
| **LNS/ALNS** | 大きく壊して直す | 局所最適が深い時、大規模問題 |
| **遺伝的アルゴリズム** | 集団の多様性+交叉 | 良い部分解の組合せで改善したい時 |
| **SWO** | 違反をフィードバック | 構築型ヒューリスティクスの改善 |

### 判断フロー

```
現在の解の問題は何か？
  ├── 制約違反がある → 修復ムーブ（ejection chain, IFS）
  ├── 局所最適に陥っている → 破壊的ムーブ（LNS, restart）
  └── もう少し改善したい → 汎用ムーブ（2-opt, swap）
        │
        汎用ムーブで改善するか？
        ├── する → SA or タブーで回す
        └── しない → 問題特化ムーブを設計する
              │
              何がボトルネックか？（②に戻る）
              → ボトルネックを直接攻めるムーブを発明
```

## ⑥ どこまで良くなり得るか？（下界・限界の把握）

### 思考

> 「今の解がどれだけ良いか、最適解にどれだけ近いか。それを知らずに最適化するのは、ゴールのないマラソンを走るのと同じ。」

### 下界の求め方

| 方法 | やり方 | 精度 |
|------|--------|------|
| **制約緩和** | きつい制約を外して解く → その解のスコアが下界 | 粗い |
| **LP緩和** | 整数制約を連続に緩和 | MIPなら精密 |
| **分解下界** | 部分問題の最適解の合計 | 問題による |
| **既知の最適解** | ベンチマークの最適解と比較 | 正確 |
| **物理的限界** | 「全員最高スキルでも上限はX」 | 粗いが直感的 |

### 判断フロー

```
下界を計算できるか？
  ├── できる → 現在の解とのギャップを計算
  │     ├── ギャップ < 1% → ほぼ最適。これ以上は無理
  │     ├── ギャップ 1-10% → 改善余地あり。ムーブを工夫
  │     └── ギャップ > 10% → 大幅に改善可能。分解やアルゴリズム変更を検討
  └── できない → ランダム/貪欲法との比較で相対評価
```

## ⑦ 何が不可能か？（不可能性の証明）

### 思考

> 「解けない問題に時間を使うな。不可能なら不可能だと証明して、問題の定義を変える提案をしろ。」

アルゴリズムの限界ではなく、問題そのものの限界を見極める。

### 不可能性の種類と対応

| 不可能性の種類 | 確認方法 | 次のアクション |
|---|---|---|
| **物理的不可能** | 数学で証明 | 問題の前提を変える提案 |
| **計算量的不可能** | NP困難性の認識 | 近似解で妥協、時間をかける |
| **手法の限界** | 大量試行で確認 | 別の手法に切り替え |

### 本プロジェクトでの実例

| 発見 | 種類 | 証明方法 | 提案 |
|---|---|---|---|
| 「6台で60箇所は時間制約を満たせない」 | 物理的不可能 | 最低ルート数の数学的計算 | 車両を増やす |
| 「貪欲法では900回中0回feasible」 | 手法の限界 | 900回の実験で確認 | SWO+IFSに切り替え |
| 「HC3違反1件は数学的に不可避（6台/15ルート）」 | 物理的不可能 | 全ペアワイズ結合を列挙 | 車両の時間帯使い回しを提案 |

### 判断フロー

```
問題が解けない
  ↓
本当に解がないのか、アルゴリズムが悪いのか？
  ├── アルゴリズムが悪い → 別の手法を試す
  │     └── 何を試しても解けない → 大量実験で「手法の限界」を確認
  └── 解が存在しない → 数学的に証明
        └── 証明できた → 「この制約下では不可能です。XXを変えれば解けます」
              → これが最も価値のある提言
```

## 思考回路の適用ガイド — ケース別

### ケースA: 新しい最適化問題に初めて取り組む

```
推奨順序: ④ → ① → ② → ⑤

  ④ パターン認識 → 「これはVRP? スケジューリング? パッキング?」
  ① 分解 → 「大きければ分ける」
  ② ボトルネック → ベースラインを動かして急所を見つける
  ⑤ ムーブ → 急所に合った対策を設計
```

### ケースB: ベースラインはあるが品質が低い

```
推奨順序: ⑥ → ② → ⑤ → ③

  ⑥ 下界 → 「あとどれだけ改善余地がある？」
  ② ボトルネック → 「何が品質を下げているか？」
  ⑤ ムーブ → 問題特化ムーブの設計
  ③ 固定 → 良い部分を固定して悪い部分だけ再最適化
```

### ケースC: 制約違反が解消できない

```
推奨順序: ⑦ → ② → ① → ⑤

  ⑦ 不可能性 → 「そもそも解があるのか？」
  ② ボトルネック → 「どの制約が壁か？」
  ① 分解 → 「制約の構造に沿って分割できないか？」
  ⑤ ムーブ → 修復ムーブ（ejection chain, IFS）
```

### ケースD: 大規模で時間内に解けない

```
推奨順序: ① → ③ → ⑤ → ④

  ① 分解 → 「空間分割? 時間分割?」
  ③ 固定 → 「確定できる変数を先に固定」
  ⑤ ムーブ → 差分計算が高速なムーブを選ぶ
  ④ パターン認識 → LNS, ALNS など大規模向け手法
```

### ケースE: コンペで上位を狙う

```
推奨順序: ④ → ⑤ → ⑥ → ③

  ④ パターン認識 → 問題の本質を見極める（Santa 2024の文章=TSP）
  ⑤ ムーブ → 問題固有のムーブを発明（これが勝敗を分ける）
  ⑥ 下界 → 改善余地を把握して効率的に攻める
  ③ 固定 → L字テクニック的な次元削減
```

### ケースF: 経営陣に改善提案を出す

```
推奨順序: ② → ⑦ → ① → ⑥

  ② ボトルネック → 「最大の問題はこの制約です」
  ⑦ 不可能性 → 「現状の条件では数学的に不可能です」
  ① 分解 → 「ここを変えれば解けます」を複数案提示
  ⑥ 下界 → 「最大XX%の改善が見込めます」

  → コスト・効果・実現性の比較表で提言
```

## まとめ: 7つの思考回路の関係

```
問題に出会う
  │
  ④ 何に似てる？（既存の武器を探す）
  │
  ① 小さくできる？（分解で扱いやすくする）
  │
  ② 急所はどこ？（ボトルネックを特定）
  │
  ├── 急所を攻める
  │     ③ 固定できる部分は固定（探索空間を縮小）
  │     ⑤ 問題に合ったムーブを設計（解を改善）
  │
  ├── 進捗を評価する
  │     ⑥ あとどれだけ良くなる？（下界と比較）
  │
  └── 行き詰まったら
        ⑦ そもそも解けるのか？（不可能性の確認）
        → 不可能なら問題定義を変える提案
```

この7つは独立ではなく、**何度も行き来する**。「ムーブを試す→ボトルネックが変わる→新しいムーブを設計」のサイクルが回る。最適化は一発で決まるものではなく、このサイクルの回転数と質で結果が決まる。
